# UC-EV-001 - Create Event

## Goal

Allow an authenticated user with a selected group context to create a new event
from the Events page.

## Scope

This use case covers:
- opening the create-event modal from the Events page,
- entering event setup data,
- creating default teams for the event,
- persisting the event through the group events API,
- updating in-memory store state so the event appears in the Events UI.

This use case does not cover:
- inviting players,
- changing invitation status,
- assigning players to teams,
- assigning trainers or shirt sets.

## Primary Actor

Authenticated User with Role trainer

## Supporting Actors

- Frontend UI (Events page and create-event modal)
- Zustand store (event mutation and sorting)
- Events API endpoint

## Preconditions

- User is authenticated.
- A group is selected in app state.
- Events page is reachable.

## Trigger

The user presses Add in the Future Events section on the Events page.

## Input Data

Required modal inputs:
- name (text)
- date (date)
- startTime (time)
- numberOfTeams (number)
- maxPlayersPerTeam (number)
- minPlayersPerTeam (number)

Optional modal inputs:
- location (text)

Default values when modal opens:
- numberOfTeams = 1
- maxPlayersPerTeam = 9
- minPlayersPerTeam = 6
- location = empty

## Main Success Scenario

1. User opens Events page and presses Add.
2. System opens Create New Event modal.
3. User enters required values and submits.
4. System trims name and location, then builds event payload.
5. System generates teams array with length numberOfTeams.
6. Each generated team is initialized with:
   - name as Team 1..N,
   - strength = 2,
   - startTime from modal,
   - selectedPlayers as empty list.
7. System sends POST request to the group events endpoint with:
   - name, date, maxPlayersPerTeam, minPlayersPerTeam, optional location,
   - teams,
   - invitations as empty list.
8. On success, returned event is appended to store and event list is sorted by date
   ascending.
9. Modal resets fields and closes.
10. Created event is visible in Events UI (Future or Past section based on date).

## Alternative Flows

### A1 - Required Data Missing

1. User submits with missing required fields.
2. Browser required constraints prevent normal submit.
3. Modal submit handler also short-circuits if name, date, or startTime is missing.
4. No API request is sent.

### A2 - API Or Network Failure During Create

1. API request fails (for example network issue, 4xx, or 5xx).
2. Store addEvent operation returns false (or throws if no selected group).
3. Failure is logged to console.
4. Current implementation closes and resets the modal regardless.
5. No new event is added to store.

### A3 - Authentication Expired

1. Events API responds 401.
2. API client clears token and redirects to login.
3. Event is not created.

## Postconditions

Success:
- Event is persisted in backend for current group.
- Event exists in frontend store and is date-sorted with existing events.
- Event has zero invitations initially.
- Event has pre-created teams based on numberOfTeams.

Failure:
- No event is added to frontend store.
- Existing events remain unchanged.
- User-visible inline error feedback in the modal is not currently provided.

## Business Rules Reflected In Current Implementation

- Event name, date, and start time are mandatory at UI level.
- Teams are created automatically during event creation.
- Team start time is initially identical for all generated teams.
- Team default strength is 2.
- minPlayersPerTeam is captured and persisted on create.
- Events are stored and displayed in chronological order.

## Validation Rules Reflected In Current Implementation

- numberOfTeams input allows range 1..10.
- maxPlayersPerTeam input allows range 1..20.
- minPlayersPerTeam input allows range 1..maxPlayersPerTeam.
- Empty location is omitted from payload.
- name and location are trimmed before payload creation.

## API Contract Used

- Endpoint: POST /api/groups/{groupId}/events
- Request body fields used by UI:
  - name
  - date
  - maxPlayersPerTeam
  - minPlayersPerTeam
  - location (optional)
  - teams (generated by UI)
  - invitations (empty array on create)
- Success response: created Event object.

## Acceptance Criteria

1. Given an authenticated user with selected group, when valid event data is
   submitted, then an event is created and appears in Events UI.
2. Given valid numberOfTeams N, when submitted, then exactly N default teams
   are created in the new event.
3. Given missing required fields, when user attempts submit, then create request
   is not sent.
4. Given create request failure, when submit occurs, then no new event is
   persisted and no new event appears in list.
5. Given a 401 API response, when submit occurs, then user is redirected to
   login by API client behavior.

## Notes

- This spec intentionally documents current behavior, including the lack of
  inline modal error feedback on create failure.
- Role-based restriction for event creation is not enforced in the current
  frontend flow beyond authentication and group context.
